# 默会知识（波兰尼）

## 概览

这个 skill 用来解释、比较和应用波兰尼意义上的默会知识。默认以中文回答，适合做概念澄清、案例拆解、理论对比，以及把“默会知识”翻译成管理、教育、产品、研究和 AI 语境中的可操作分析。

## 触发场景

当用户要做这些事情时，使用本 skill：

- 解释“默会知识”“缄默知识”“隐性知识”是什么
- 解释波兰尼、`we know more than we can tell`、具身认知、熟练判断
- 比较默会知识与显性知识、程序性知识、陈述性知识、规则手册
- 分析学徒制、手艺、经验、判断力、专家能力如何传承
- 把默会知识应用到企业管理、组织学习、人才培养、产品设计、科研训练、AI
- 让某个文本、案例、组织问题从“波兰尼视角”重新解释
- 生成中文讲义、提纲、讨论题、课堂案例、培训模板

如果用户没有指定语言，优先中文；如果用户明确要英文或学术英文，再切换。

## 默认工作流

### 1. 先判断用户要哪一种回答

优先识别用户需求属于哪一类：

- 快速定义
- 深入解释
- 对比分析
- 案例拆解
- 领域应用
- 文本解读
- 教学材料

如果用户要的是实务答案，不要一上来铺太多哲学史背景。

### 2. 抓住核心命题

回答时围绕这个核心：

- 人的很多“会做”“会判断”“会识别”的能力，不能被完整还原成可明说的规则

要同时保持几个边界：

- 默会知识不等于神秘主义
- 默会知识不等于“只是没写文档”
- 默会知识和显性知识通常是互补关系，不是非此即彼
- 熟练能力往往依赖情境、练习、模仿、反馈、身体感、注意力分配与判断

### 3. 选合适的表达框架

按用户问题切换下面的输出框架：

- **定义框架**：一句定义 + 为什么重要 + 一个例子
- **对比框架**：默会知识 vs 显性知识，按表达、传递、验证、局限来比
- **案例框架**：指出活动中的默会部分，说明为什么不能完全手册化
- **应用框架**：推导出培训、招聘、带教、文档、组织设计、AI 使用上的含义
- **文本框架**：区分核心观点、常见误解、当代应用

### 4. 尽量用具体例子，不停留在抽象层

优先使用这类例子：

- 骑自行车
- 医生做模式识别式诊断
- 工匠通过手感调整力度和角度
- 资深工程师识别架构风险
- 代码评审中超出 checklist 的质量判断
- 产品经理或设计师的“产品感”“审美判断”
- 老师带学生、师傅带徒弟

### 5. 提炼成实践含义

如果用户问应用层，优先落到这些结论：

- 文档很重要，但文档不足以传完能力
- 培养能力需要观察、跟做、纠偏、重复练习
- 专家知识转移往往依赖陪练、搭档、复盘、示范
- 组织如果只依赖少数人的脑内 know-how，会有脆弱性
- AI 可以复现模式，但不应轻率等同于人类具身而负责的熟练判断

如果某个结论带有强解释性或争议，明确标注为“这是解释”或“这是较强主张”。

## 中文输出模板

### 快速定义模板

1. 先给一句中文定义
2. 再说它与显性知识的区别
3. 最后给一个生活或工作例子

### 对比模板

- 表达方式
- 传递方式
- 是否依赖情境
- 是否依赖练习
- 过度规则化时会损失什么

### 案例拆解模板

1. 这项活动要完成什么
2. 哪些部分可以文档化
3. 哪些部分依赖经验、感觉、判断或时机
4. 为什么这些部分难以完全显性化
5. 如果要传承，应该怎么设计机制

### 领域应用模板

1. 点出该领域最关键的默会部分
2. 说明它为什么抗拒完全规则化
3. 给出实际建议

## 常用中文术语

- 默会知识
- 缄默知识
- 隐性知识
- 显性知识
- 具身技能
- 实践性 know-how
- 经验判断
- 学徒制
- 技能传承
- 组织学习

如果用户明显在学术语境中，优先使用“默会知识”；若用户来自企业管理语境，可同时提到“隐性知识”。

## 质量标准

- 不要把默会知识简化成“没有 SOP 的东西”
- 不要把默会知识和显性知识讲成完全割裂
- 不要不加限定地把 AI、创造力、意识直接并入同一个论断
- 用户要 practical answer 时，先给应用，再补理论
- 用户要学术风格时，再提升术语密度和论证层次
